# Security Architecture

This document describes how sigyn protects stored secrets, the trust boundaries in the system, and what the project was **not** designed to defend against.

## Threat Model

sigyn is a **local-first, single-user** tool. It assumes:

- the operator is the device owner
- the device has macOS FileVault enabled (full-disk encryption)
- the macOS user account is protected by a strong password
- no other user on the machine is actively hostile

The primary goal is to keep environment secrets encrypted at rest so they are not exposed by accidental file sharing, backup leaks, or casual filesystem access.

## Encryption

### Algorithm

All entry values are encrypted with **AES-256-GCM-SIV** (RFC 8452), a nonce-misuse-resistant authenticated encryption scheme. Even if a nonce were accidentally reused, the only information leaked would be whether two plaintexts are identical — there is no catastrophic key recovery as with standard AES-GCM.

### Key Hierarchy

```
macOS Keychain
  └─ master key (256-bit random, generated once)
       └─ per-value encryption (AES-256-GCM-SIV)
            ├─ random 96-bit nonce per encrypt call
            └─ ciphertext + auth tag stored as base64 in SQLite
```

- The master key is a 32-byte random value generated by `rand::random()` (ChaCha-based CSPRNG).
- Each encrypted value stores its own random nonce prepended to the ciphertext, then base64-encoded.
- There is one master key per installation. Rotating it requires re-encrypting all stored values (not yet implemented).

### What Is Encrypted

| Data | Encrypted? |
|---|---|
| Entry values (the actual secrets) | Yes — AES-256-GCM-SIV |
| Project names | No |
| Entry names (env var keys) | No |
| Categories and descriptions | No |
| Environment labels | No |
| Project structure and relationships | No |

An attacker with filesystem access to the SQLite database can see which projects exist, what env var keys are defined, and how they are organized — but cannot read the actual secret values without the master key.

## Authentication

### Device Authentication

Unlocking sigyn requires **macOS LocalAuthentication** (`LAPolicy::DeviceOwnerAuthentication`), which means:

- Touch ID if enrolled
- device password as fallback
- no custom password, no cloud auth, no API keys

This delegates authentication entirely to the operating system. sigyn never sees or stores the user's password or biometric data.

### Session Management

After a successful unlock:

1. The master key is loaded from Keychain into an in-memory session cache.
2. Every operation that touches secrets refreshes a last-activity timestamp.
3. After **5 minutes of inactivity**, the session expires automatically.
4. On expiry (or manual lock), the cached key is **zeroized** in memory before being dropped.
5. A background thread checks for idle expiry every 15 seconds.

```
Unlock (Touch ID / password)
  → Load master key from Keychain
  → Cache in memory (Vec<u8>)
  → 5 min idle timeout
  → Zeroize key bytes on lock/expiry
```

## Key Storage

The master key is stored in the **macOS Keychain** as a generic password item:

- **Service:** `sigyn`
- **Account:** `master-key`
- **Access:** governed by Keychain ACLs (default: accessible when device is unlocked)

The key is never written to disk, environment variables, config files, or logs.

### First-Run Behavior

On first unlock, if no keychain entry exists and the database has no encrypted values, sigyn generates a new random master key and stores it in Keychain. If encrypted values already exist but the keychain entry is missing (e.g., after a Keychain reset), sigyn refuses to proceed and displays an error rather than silently generating a new key that cannot decrypt existing data.

## File System Permissions

| Path | Permissions | Contents |
|---|---|---|
| `~/.local/share/com.connorguy/sigyn/` | `0o700` | Data directory |
| `~/.local/share/com.connorguy/sigyn/sigyn.sqlite3` | `0o600` | SQLite database |
| `~/.local/share/com.connorguy/sigyn/cli.sock` | `0o600` | IPC Unix socket |

All paths are restricted to the owning user. The parent directory's `0o700` permission prevents other local users from listing or accessing any contents.

## IPC (CLI ↔ Desktop)

The desktop app starts a Unix domain socket server at the path above. Currently the protocol supports only a **Ping / Pong** liveness check — no secrets are transmitted over IPC.

Safeguards:

- Socket file permissions: `0o600`
- Parent directory permissions: `0o700`
- Read timeout: 5 seconds
- Maximum request size: 64 KB
- JSON-only protocol (no shell evaluation)

### CLI Authentication

The companion CLI authenticates **independently** — it triggers its own macOS LocalAuthentication prompt and reads the master key from Keychain directly. It does not request secrets from the desktop app over IPC. Both the desktop app and the CLI have equal access to the Keychain entry and the SQLite database.

## Frontend Security

### Content Security Policy

```
default-src 'self';
img-src 'self' asset: https://asset.localhost;
style-src 'self' 'unsafe-inline'
```

- No external scripts, fonts, or resources are loaded.
- No `eval()`, `innerHTML`, or dynamic code execution.
- `unsafe-inline` for styles is required by the React rendering pipeline; the app loads no external content, so the risk surface is minimal.

### Tauri Capabilities

The app requests only `core:default` permissions — no filesystem, shell, HTTP, or notification capabilities are granted to the webview.

### Input Handling

- All SQL queries use parameterized statements (`params![]` macro) — no string interpolation.
- Project names, entry names, and environment labels are normalized and validated before storage.
- Entry names are uppercased, stripped to `[A-Z0-9_]`, and trimmed of leading/trailing underscores.

## CLI Safety

### Restricted Environment Variables

`sigyn run` refuses to inject values for dangerous environment variable names that could alter process loading or execution behavior:

```
PATH, IFS, ENV, BASH_ENV, GCONV_PATH,
JAVA_TOOL_OPTIONS, NODE_OPTIONS,
PERL5LIB, PERL5OPT, PYTHONHOME, PYTHONPATH,
RUBYLIB, RUBYOPT,
LD_* (prefix), DYLD_* (prefix)
```

If any entry name matches this blocklist, the command exits with an error before launching the child process.

### Process Injection

Environment variables are injected using Rust's `Command::env()` API, which sets variables directly on the child process without shell evaluation. There is no intermediate shell, no `eval`, and no `.env` file written to disk.

## What sigyn Does NOT Protect Against

### Full Device Compromise

If an attacker has root access or can run code as your user, they can:

- read the Keychain entry (after passing device auth, which they control)
- attach a debugger to the running process and read the cached key from memory
- modify the sigyn binary to exfiltrate secrets

sigyn is not a defense against a compromised operating system.

### Memory Forensics

While the master key is zeroized on lock and expiry, Rust's allocator may leave copies of sensitive data in freed heap pages, and the OS may swap memory contents to disk. sigyn does not use `mlock` or other memory-locking mechanisms. On macOS with FileVault enabled, swap is encrypted, which mitigates this.

### Metadata Exposure

Project names, entry names (env var keys), categories, descriptions, and environment labels are stored as **plaintext** in the SQLite database. An attacker with filesystem access can see the structure and naming of all secrets without the master key.

### Multi-User or Networked Environments

sigyn has no concept of teams, shared vaults, access control lists, or audit logs. It is designed for a single developer on a single machine. Sharing secrets between users requires exporting and re-importing outside of sigyn.

### Key Rotation

There is currently no built-in mechanism to rotate the master key. If you suspect the key has been compromised, the current remediation is:

1. Export all secrets (preview/copy)
2. Run `sigyn reset-test-data --yes` to destroy the database and keychain entry
3. Relaunch and re-import

### Backup and Recovery

If the macOS Keychain entry is lost (e.g., after a Keychain reset or migration to a new machine without Keychain transfer), all encrypted values become permanently unrecoverable. There is no key escrow, recovery phrase, or backup mechanism.

### Clipboard Security

When using the "Copy" button in the desktop app, decrypted secrets are placed on the system clipboard in plaintext. Any application with clipboard access can read them. sigyn does not clear the clipboard after a timeout.

### Side-Channel Attacks

The encryption implementation relies on the `aes-gcm-siv` Rust crate, which uses hardware AES-NI instructions where available. No specific countermeasures are taken against timing attacks on key comparison or cache-based side channels beyond what the underlying crate provides.

## Dependencies

Security-critical Rust crates:

| Crate | Purpose |
|---|---|
| `aes-gcm-siv` | AES-256-GCM-SIV encryption/decryption |
| `rand` | Cryptographic random number generation (nonces, master key) |
| `zeroize` | Secure memory clearing for key material |
| `security-framework` / `security-framework-sys` | macOS Keychain access |
| `objc2-local-authentication` | macOS Touch ID / device password |
| `rusqlite` (bundled) | SQLite with parameterized queries |
| `tauri` | Application framework with CSP and capability sandboxing |
